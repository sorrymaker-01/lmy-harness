## 工作流决策树

**这是什么类型的任务？**

```
┌─ 用源数据填充空白模板？
│  └─→ 遵循下方的"模板填充工作流"
│
├─ 编辑已填充的现有幻灯片？
│  └─→ 提取当前内容，修改，重新验证
│
└─ 修复现有幻灯片上的格式问题？
   └─→ 参见"常见失败"表，应用针对性修复
```

## ⚠️ 关键渲染限制

**LibreOffice用于验证但无法准确渲染PowerPoint文件。** 它会破坏字体、渐变、形状位置、文本换行和某些表格格式。

**这意味着：** 在LibreOffice中通过视觉验证的幻灯片在Microsoft PowerPoint中可能仍有问题。验证循环捕获结构性问题（缺失内容、表格损坏、保留占位符格式），但**无法**捕获字体替换、细微对齐偏移或渐变问题。

**必需操作：** 交付输出时始终包含此声明：
> "此文件使用LibreOffice验证。请在分发前在Microsoft PowerPoint中审阅，因为可能存在渲染差异。"

## 模板填充工作流

复制并跟踪进度：

```
演示文稿进度：
- [ ] 阶段1：提取并验证源数据
- [ ] 阶段2：将内容映射到模板部分
- [ ] 阶段3：以正确格式填充幻灯片
- [ ] 阶段4：验证 → 修复 → 重复直到干净
- [ ] 阶段5：最终验证
```

### 阶段1：数据提取
1. 在任何修改之前**创建原始模板的备份** — 复制到`[文件名]_backup.pptx`。直接XML编辑或意外错误可能损坏文件。
2. 识别所有源材料（Excel、CSV、PDF报告、Word文档、数据库、网络来源）
3. 从每个来源提取相关数据点
4. 根据原始来源验证所有数字
5. 标准化单位和货币（将所有数字转换为模板中使用的主要单位/货币）
6. 注意任何需要验证的计算 → 参见[`calculation-standards.md`](reference/calculation-standards.md)获取公式

### 阶段2：内容映射
1. **打开并视觉审查模板** — 在修改前了解其结构、风格和现有内容
2. 分析模板结构 — 识别所有占位符区域和内容框
3. 将源数据映射到相应的模板部分 → 参见[`slide-templates.md`](reference/slide-templates.md)获取映射指导
4. 识别占位符指导框（任务创建者的彩色说明框）
5. 注意任何数据缺口或不匹配 → 参见[`slide-templates.md`](reference/slide-templates.md#handling-data-template-mismatches)获取解决方案

### 阶段3：模板填充
1. **删除或重新格式化占位符框** — 彩色说明框显示要创建什么内容，而非如何格式化。删除它们并在其位置创建正确格式化的内容。参见[关键反模式](#critical-anti-patterns-never-do-these)。
2. 用映射的内容填充每个部分（首先关注内容）
3. **然后应用格式化**以匹配模板风格 → 参见[`formatting-standards.md`](reference/formatting-standards.md)
4. 将表格创建为实际表格对象（绝不使用管道符/制表符分隔的文本） → 参见[`xml-reference.md`](reference/xml-reference.md#table-implementation)
5. 将箭头/形状创建为PowerPoint对象 → 参见[`xml-reference.md`](reference/xml-reference.md#arrow-shapes)
6. 如果任务文件中提供了公司标志则插入；如果不可用，标记给用户："[未提供标志 - 请提供公司标志]"

### 阶段4：验证 → 修复 → 重复

**这是一个反馈循环。重复直到所有检查通过或触发升级。**

```bash
# 转换为图像进行视觉验证
soffice --headless --convert-to pdf presentation.pptx
pdftoppm -jpeg -r 150 presentation.pdf slide
```

**验证清单（检查每张幻灯片图像）：**
- [ ] 文本在背景上可读？
- [ ] 表格是实际对象（列对齐，不是管道符/制表符分隔的文本）？
- [ ] 图表/表格填满指定区域？
- [ ] 项目符号格式在各部分内一致？
- [ ] 字体大小在同级框中匹配？
- [ ] 没有内容超出幻灯片边界？
- [ ] **没有保留占位符格式**（没有大彩色框内堆砌数据）？
- [ ] **没有基于文本的"表格"**（没有`|`或制表符分隔符创建假列）？
- [ ] **跨幻灯片一致性**：相同指标/数字在所有出现的幻灯片上完全相同？

**修复循环协议：**

| 循环 | 操作 |
|-------|--------|
| 1 | 修复所有已识别问题，重新验证 |
| 2 | 修复剩余问题，重新验证 |
| 3 | 如果问题持续存在，记录剩余问题并升级给用户 |

**3次循环后，如果问题仍然存在：**
1. 列出每个未解决的问题，包括幻灯片编号和描述
2. 解释尝试了什么
3. 交付文件并明确声明："以下问题无法自动解决：[列表]。需要手动审查。"

**不要**无限循环。某些问题（字体渲染、复杂形状对齐）可能需要在PowerPoint中手动干预。

### 阶段5：最终验证

在交付前运行[最终质量清单](#final-quality-checklist)。

| 上下文 | 符号 | 用法 |
|---------|--------|-------|
| 包含/正面 | ✓ | 范围内的项目，存在的功能 |
| 排除/负面 | × | 范围外的项目，不存在的功能 |
| 中性列表 | • | 一般枚举，评论 |
| 编号序列 | 1. 2. 3. | 流程步骤，排名 |
| 子项目符号 | – | 主要项目符号下的次要要点 |

### 幻灯片层级级别（典型）

这些是典型范围 — 根据模板规格调整：

| 级别 | 示例 | 典型大小 | 风格 |
|-------|----------|--------------|-------|
| 标题 | 幻灯片标题 | 40-48pt | 粗体 |
| 副标题 | 市场定义，幻灯片描述符 | 18-22pt | 粗体 |
| 部分标题 | "关键预测"，"评论" | 14-16pt | 常规 |
| 区块标签 | "包含的细分"，"定义"侧边栏 | 12-14pt | 常规 |
| 区块内容 | 项目符号，正文 | 11-14pt | 常规 |
| 表格标题 | 列标题 | 10-12pt | 粗体 |
| 表格正文 | 单元格内容 | 9-11pt | 常规 |
| 脚注 | 来源，注释 | 8-9pt | 斜体 |

### 字体一致性匹配

**相同层级级别**的框必须使用相同的字体大小：

| 相同级别 | 必须匹配 |
|------------|-----------------|
| "包含的细分" | "排除的细分" |
| "定义" | "范围理由" |
| 左列项目符号 | 右列项目符号 |
| 所有区块标签 | 彼此 |
| 所有部分标题 | 彼此 |

### 演示文稿舍入

这些是**典型约定** — 根据数值大小和模板风格调整：

| 值类型 | 典型舍入 | 示例 |
|------------|------------------|---------|
| 大市场规模（$100亿+） | 最接近$10亿 | 18.5 → $190亿 |
| 较小市场规模（<$100亿） | 最接近$5亿或$1亿 | 2.3 → $25亿 |
| 规模范围 | 匹配来源精度 | 14.9-22.1 → $150-220亿 |
| CAGR | 整数%或0.5% | 16.4% → 16%或16.5% |
| 市场份额 | 最接近5%或匹配来源 | 21.4% → 20% |
| 倍数 | 1位小数 | 9.69 → 9.7x |

**原则：** 舍入不应显著改变数值。对于较小值，使用更精细的精度。

### 文本密度规则

- 每个内容框最多6-7个项目符号
- 每个项目符号最多2行
- 括号示例：同一行或在下方缩进
- 没有孤立词（新行上只有一个词）

### 对齐原则

**垂直堆叠的框**必须具有相同的：
- 左边距位置，项目符号缩进，文本起始位置，框宽度

**水平相邻的框**必须具有相同的：
- 顶部位置，高度（如可能），内部填充

### 多幻灯片一致性

当相同数据出现在多张幻灯片上时：
- 使用相同的数字、格式和术语
- 如果一张幻灯片上的指标更新，更新所有出现的地方
- 在验证期间交叉引用以发现不匹配

## 必需要求

无论模板如何，这些要求都是不可协商的：

| 要求 | 详情 |
|-------------|---------|
| **文本可读性** | 所有文本必须与背景有足够对比度。示例：深蓝、深绿、黑色背景上的白色/浅色文本；白色、浅灰、浅黄色背景上的黑色/深色文本。 |
| **实际表格对象** | 表格数据必须是表格对象，而非制表符分隔文本。参见[`xml-reference.md`](reference/xml-reference.md#table-implementation)。 |
| **正确的图表/表格大小** | 粘贴的视觉元素必须填满指定区域。参见[`formatting-standards.md`](reference/formatting-standards.md#chart-and-image-handling)。 |
| **一致的格式** | 部分内的项目符号必须匹配（符号、大小、缩进）。同级框必须使用相同字体大小。 |
| **内容边界** | 所有内容必须保持在幻灯片边缘内。脚注框宽度：16:9约32.5cm，4:3约24cm。 |
| **无占位符格式** | 删除彩色说明框。主体内容：根据模板使用浅色背景上的深色文本。 |

## 关键反模式：绝不要这样做

当占位符格式被误认为是输出格式时会发生这些失败。识别这些模式至关重要。

### 反模式1：将数据填充到占位符框中

**发生的情况：** 模板有彩色说明框（黄色、橙色等），带有指导文本。模型用实际数据替换指导文本，但保留了彩色框。

**为什么错误：** 彩色框就是占位符。它告诉你内容放在哪里。输出应该有不同的格式 — 通常是浅色背景上的深色文本，或正确样式的形状。

**识别测试：** 如果你填充的幻灯片有大彩色矩形填充数据文本，你复制了占位符格式而不是替换它。

**关键区别 — 两种类型的"占位符"：**

| 类型 | 如何识别 | 做什么 |
|------|-----------------|------------|
| **说明框** | 鲜艳颜色（黄色、橙色），包含指导文本如"在此插入X"，彩色背景上的白色/浅色文本 | 删除整个形状，然后用生产格式创建新内容 |
| **布局占位符** | 幻灯片母版/布局的一部分，与模板主题匹配的中性颜色，"单击添加文本" | 保留形状，仅替换文本内容 |

如果不确定：检查来自同一模板的空白幻灯片上是否存在该形状。布局占位符持续存在；说明框是常规形状。

### 反模式2：基于文本的"表格"

**发生的情况：** 模型使用分隔符字符（`|`、制表符、空格）而不是实际表格对象创建类似表格的内容。

**为什么错误：** 这不是表格。列永远无法正确对齐，无法一致格式化，看起来不专业。

**识别测试：** 如果你正在输入`|`字符或依赖空格/制表符创建列，你正在创建文本，而不是表格。

**必须验证：** 创建任何表格后，验证它是实际的表格对象。参见[`xml-reference.md`](reference/xml-reference.md#critical-verify-tables-are-actual-table-objects)获取验证方法。

### 反模式3：继承占位符对比度

**发生的情况：** 占位符使用彩色背景上的浅色文本（例如，黄色上的白色）。模型填充数据但保留此配色方案，导致难以阅读的输出。

**为什么错误：** 占位符颜色故意独特以表示"替换我"。生产幻灯片通常在主体内容区域使用浅色背景上的深色文本。

**识别测试：** 如果你填充的内容在主体区域（非标题）的鲜艳彩色背景上有浅色/白色文本，你继承了占位符格式。

**正确方法：** 应用生产格式 — 通常主体内容使用浅色背景上的深色文本（#000000或#333333）。标题和强调区域可以使用品牌颜色。

### 总结：占位符 vs. 生产

| 元素 | 占位符（输入） | 生产（输出） |
|---------|---------------------|---------------------|
| 说明框 | 彩色背景，指导文本 | 删除或重新格式化 |
| 数据区域 | "[在此插入数据]"文本 | 带有干净格式的实际数据 |
| 表格 | 表格应包含内容的描述 | 带有行/列的实际表格对象 |
| 正文 | 彩色背景上的浅色文本 | 浅色背景上的深色文本 |

**占位符告诉你创建什么内容，而非如何格式化。**

## 错误处理

**如果PDF/图像转换失败：**
1. 检查LibreOffice是否安装：`which soffice`
2. 尝试替代方案：`libreoffice --headless --convert-to pdf presentation.pptx`
3. 如果仍然失败，在PowerPoint/LibreOffice中手动打开并导出

**如果源数据有不一致或冲突：**
1. **优先顺序**：首先使用任务文件中明确提供的数据
2. 如果使用其他来源的数据（网络搜索、外部文档），标记给用户
3. 明确记录任何差异
4. 添加脚注解释数据来源选择

**如果计算与来源预测不匹配：**
1. 展示你的计算方法论
2. 注意差异和可能原因（不同基准年、方法论）
3. 如果差异重大，同时呈现两个值
4. 标记给用户解决

## 表格结构指南

创建表格时（必须是实际表格对象）：

**列对齐：**
- 文本列：左对齐（标题和内容）
- 数字列：右对齐或居中对齐（标题与内容匹配）

**标题行：**
- 粗体文本
- 阴影背景（模板的品牌颜色）
- 白色或对比色文本

**一致预期/总计行：**
- 粗体文本
- 上方分隔线
- 明显的背景阴影

**宽度：** 完全填满指定部分宽度。

有关XML实现，参见[`xml-reference.md`](reference/xml-reference.md#table-implementation)。

## 脚注格式

**格式：**
```
来源：[来源1]（年份），[来源2]（年份）。
注释：(1) [第一条注释]；(2) [第二条注释]。
```

**示例：**
```
来源：Grand View Research（2024），Mordor Intelligence（2024），Markets and Markets（2023）。
注释：(1) 不包括硬件收入；(2) 包括B2B和B2C细分市场。
```

幻灯片正文中的所有上标数字（¹、²、³）必须有对应的注释条目。

## 最终质量清单

在交付填充的模板之前，验证：

### 数据准确性
- [ ] 所有数字与原始来源文档匹配
- [ ] 计算值根据公式验证（参见[`calculation-standards.md`](reference/calculation-standards.md)）
- [ ] 年份和时间段正确
- [ ] 公司/竞争对手名称拼写正确
- [ ] 相同数字在所有出现的幻灯片上完全相同

### 内容映射
- [ ] 每个模板部分填充了适当的数据
- [ ] 没有剩余的`[括号]`占位符文本
- [ ] 所有来源引用包含在脚注中
